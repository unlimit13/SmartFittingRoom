# 교육생 프로젝트 산출물 가이드

이 가이드는 팀 프로젝트가 **평가 Agent와 평가자(강사)에 의해 정확하게 평가**될 수 있도록,
제출 산출물과 형식을 안내합니다.

이 프로젝트는 **요구사항 분석부터 구현·테스트·검증까지 개발의 전 과정을 팀이 스스로 완결**하는
1인 완결형(Full Cycle) 개발을 지향합니다. 따라서 산출물도 그 전 과정이 드러나야 하며,
**요구사항은 주어지는 것이 아니라 팀이 직접 분석하여 명세**합니다.

> 형식을 지키지 않으면 자동 분석이 제한되어 평가에 불리할 수 있습니다.
> 형식을 맞추는 것 자체가 "내 결과물을 남이 검증할 수 있게 만든다"는 실무 역량입니다.

## 1. 제출 산출물 (필수)

평가 대상은 **반별·팀별 폴더**이며, 아래 6종을 제출합니다.

| 산출물 | 파일명 형식 | 내용 |
|---|---|---|
| 요구사항 명세서 | `LG_부트캠프_13기_X반_요구사항_명세서(X팀).md` | 팀이 직접 분석·정의한 요구사항 + 검증 기준 + 시연 타임스탬프 (**2번**) |
| 결과보고서 | `LG_부트캠프_13기_X반_결과보고서(X팀).pdf` **및** `.pptx` | 프로젝트 내용·설계·결과 정리 (PDF·PPT 두 가지 형식 모두 제출) |
| 결과파일 | `LG_부트캠프_13기_X반_결과파일(X팀).zip` | 소스코드·테스트·실행 자원·Git 히스토리 (**3번**) |
| 시연영상 | `LG_부트캠프_13기_X반_시연영상(X팀).mp4` | 요구사항별 동작 시연 (**5번**) |
| 포스터 | `LG_부트캠프_13기_X반_포스터(X팀).pdf` **및** `.pptx` | 프로젝트 요약 (PDF·PPT 두 가지 형식 모두 제출) |
| 평가 결과 리포트 | `LG_부트캠프_13기_X반_에이전트(X팀).html` | 본 평가 Agent를 직접 실행해 생성한 자체 평가 결과 (**8번**) |

> `X반`·`X팀`은 실제 반/팀 번호로 바꿔 작성하세요. (예: `LG_부트캠프_13기_A반_요구사항_명세서(4팀).md`)

폴더 구조 예시:
```text
LG_부트캠프_13기_X반_최종결과물(X팀)/
├── LG_부트캠프_13기_X반_요구사항_명세서(X팀).md
├── LG_부트캠프_13기_X반_결과보고서(X팀).pdf
├── LG_부트캠프_13기_X반_결과보고서(X팀).pptx
├── LG_부트캠프_13기_X반_결과파일(X팀).zip
├── LG_부트캠프_13기_X반_시연영상(X팀).mp4
├── LG_부트캠프_13기_X반_포스터(X팀).pdf
├── LG_부트캠프_13기_X반_포스터(X팀).pptx
└── LG_부트캠프_13기_X반_에이전트(X팀).html
```

## 2. 요구사항 명세서 (`..._요구사항_명세서(X팀).md`)

Full Cycle 개발의 출발점입니다. **과제를 팀이 스스로 분석하여 요구사항을 정의**하세요.
요구사항은 **기능/비기능을 구분하지 않고** 하나의 목록으로 작성합니다. (비기능 요소는 Agent가 추출합니다.)

각 요구사항에는 아래 항목을 함께 적습니다.

- **검증 기준(완료 조건)** — 무엇을 충족하면 "완료"인지 (측정 가능하게)
- **시연 타임스탬프** — 해당 기능이 시연 영상의 몇 분 몇 초에 나오는지

| 요구사항 ID | 요구사항 내용 | 검증 기준 (완료 조건) | 시연 타임스탬프 |
|---|---|---|---|
| R-01 | 웹에서 실시간 카메라 영상이 표시된다 | 브라우저에서 끊김 없이 피드가 보인다 | 00:12 |
| R-02 | 사람 등장 시 자동 캡처된다 | 사람 등장 후 캡처 이미지가 생성된다 | 00:45 |
| R-03 | 검색 결과는 2초 이내 표시된다 | 동일 조건 측정 시 평균 ≤ 2초 | N/A (영상 미시연) |
| ... | ... | ... | ... |

> - 요구사항 ID는 **기능/비기능을 구분하지 않는 중립 번호(`R-01`, `R-02` …)** 를 사용하세요. (`FR`/`NFR`처럼 미리 분류하는 접두어 금지)
> - 분류는 Agent가 합니다 — 기능 요구사항은 코드·테스트와 매핑하고, 비기능 요구사항은 평가 카드로 추출합니다.
> - 검증 기준이 구체적·측정 가능할수록 자동 검증과 평가 정확도가 올라갑니다.
> - 영상으로 보여줄 수 없는 요구사항은 타임스탬프 대신 `N/A (사유)`로 적으세요. 비어 있으면 `시연 미입증`으로 처리됩니다.

## 3. 결과파일 (`..._결과파일(X팀).zip`) 구성

자동 분석(기능 구현·테스트·실행 가능성·협업)이 이뤄지는 핵심 자원입니다. 아래를 포함하세요.

```text
결과파일.zip
├── src/                 # 소스 코드
├── tests/               # 테스트 코드
├── test-results/        # (실행 불가 시) 테스트 실행 결과 로그·리포트
├── README.md            # 프로젝트 개요
├── RUN.md               # 실행·재현 문서 (4번의 8개 기준 충족)
├── 라이브러리 목록 파일   # requirements.txt / package.json 등 (버전 고정)
└── .git/ 또는 git_log.txt # Git 히스토리 (6번)
```

> `.zip` 안에 Git 히스토리(`.git` 폴더 또는 `git log` export)가 없으면 협업 분석은 생략되고 그 사실이 리포트에 표시됩니다.

**프로젝트 내 프로그램이 2개 이상인 경우**, `src/`·`tests/`·`test-results/` 아래를 프로그램별로 나눠 구성하세요. 디렉토리 이름은 `program_역할` 형식으로 짓습니다. (예: `program_frontend`, `program_backend`)

```text
결과파일.zip
├── src/
│   ├── program_역할_1/   # 첫 번째 프로그램 소스 코드 (예: program_frontend)
│   └── program_역할_2/   # 두 번째 프로그램 소스 코드 (예: program_backend)
├── tests/
│   ├── program_역할_1/   # 첫 번째 프로그램 테스트 코드
│   └── program_역할_2/   # 두 번째 프로그램 테스트 코드
├── test-results/
│   ├── program_역할_1/   # 첫 번째 프로그램 테스트 실행 결과
│   └── program_역할_2/   # 두 번째 프로그램 테스트 실행 결과
├── README.md
├── RUN.md
├── 라이브러리 목록 파일
└── .git/ 또는 git_log.txt
```

## 4. 실행·재현 문서 (`RUN.md`) — DevOps 8개 기준

평가 Agent는 "제출물만으로 다시 띄울 수 있는가"를 아래 **8개 기준**으로 확인합니다.
`RUN.md`(또는 README)에 아래가 모두 담기도록 작성하세요.

1. 사용한 라이브러리(패키지) 목록이 명시되어 있는가 (`requirements.txt`, `package.json` 등)
2. 라이브러리 버전이 고정되어 있는가
3. 빌드·설치 방법이 문서화되어 있는가
4. 실행·기동 방법이 문서화되어 있는가
5. 환경 변수·설정 파일이 문서화되어 있는가 (`.env` 예시 등)
6. 외부 서비스·자원(DB, API 키 등)이 안내되어 있는가
7. 빌드·설치가 문서대로 성공하는가
8. 애플리케이션이 실행·기동되는가

예시:
```bash
# 1~2. 라이브러리 설치 (버전 고정)
pip install -r requirements.txt
# 3~4. 빌드/실행
python app.py        # http://localhost:8000
# 5~6. 환경설정: .env.example 참고, GEMINI_API_KEY 등 외부 키 설정 안내
```

## 5. 시연 영상 (`..._시연영상(X팀).mp4`)

- **모든 요구사항을 영상에 담을 필요는 없습니다.** 핵심 기능을 중심으로, 영상으로 보여줄 수 있는 요구사항 위주로 녹화하세요.
- **영상에 담은 요구사항은 동작 장면이 명확히 드러나도록** 녹화하세요.
- 각 요구사항이 나오는 지점을 **요구사항 명세서의 시연 타임스탬프 칸**에 기입하세요. (2번)
- Agent는 영상을 직접 판정하지 않고, 타임스탬프 기반 **확인표**를 만들어 평가자에게 전달합니다.
  실제 동작 여부는 평가자가 해당 지점을 보고 판단합니다.
- 영상으로 보여줄 수 없는 요구사항은 `N/A (사유)`로 표기하세요.

## 6. 테스트

- 각 요구사항의 **검증 기준을 확인하는 테스트**를 작성하세요.
- 테스트는 **자동 실행 가능**하도록 구성하세요.
- 평가 환경에서 실행이 어려운 경우를 대비해, **테스트 실행 결과(로그·리포트)** 를 `test-results/`에 함께 첨부하세요.
  (Agent는 실행 가능하면 통과/실패를 직접 확인하고, 불가하면 첨부된 결과를 확인합니다.)

**코드·테스트·결과가 서로 연결되도록** 아래 관례를 지키세요. (특별한 추가 작업이 아니라, 표준 테스트 작성 관례를 따르면 자연스럽게 연결됩니다.)

- **배치** — 테스트 디렉토리(`tests/`)를 소스(`src/`)와 같은 구조로 두세요. 예: `src/search/query.py` → `tests/search/test_query.py`
- **네이밍** — 프레임워크 표준 패턴을 쓰세요. 예: `test_*.py`(pytest), `*.test.js`/`*.spec.ts`(Jest), `*Test.java`(JUnit). 테스트 이름은 무엇을 검증하는지 드러나게 짓습니다.
- **결과 연결** — 테스트 실행 결과는 표준 리포트 포맷(JUnit XML, pytest `--junitxml`, Jest 결과 등)으로 남기면 좋습니다. 결과 파일 안에 테스트 이름·통과/실패가 들어 있어 테스트 코드와 자동으로 매칭됩니다.

> 위 배치·네이밍만 지켜도 "이 코드 ↔ 이 테스트 ↔ 이 결과"가 자동으로 드러납니다. 요구사항 ID를 테스트 이름에 넣으면(예: `test_R03_search_under_2s`) 요구사항까지 한 줄로 추적됩니다.

## 7. Git 사용 (협업 분석 대상)

협업은 Git 기록으로 평가됩니다. **팀 전원의 참여**가 기록에 남도록 하세요.
Agent가 보는 핵심 신호는 다음 3가지입니다.

- **기여 분산도** — 팀원 전원이 참여했는가, 특정 1인에 집중되지 않았는가
- **커밋 지속성** — 2주간 꾸준히 작업했는가, 마감 직전에 몰리지 않았는가
- **협업 상호작용** — PR/리뷰·머지 등 함께 작업한 기록이 있는가

권장:
- 각자 **본인 계정**으로 커밋 (author 정보 정확히)
- 작업을 꾸준히 분산 (막판 몰아치기 지양)
- PR/리뷰·머지로 함께 작업한 흔적 남기기
- **Git 히스토리를 결과파일.zip에 포함** (`.git` 폴더 또는 `git log` 결과 파일)

> 평가는 개인 기여량 순위가 아니라 **팀 전체가 협업했는지**를 봅니다.

## 8. 자체 평가 결과 리포트 (`..._에이전트(X팀).html`)

제출 전에 **본 평가 Agent를 팀 프로젝트에 직접 실행**하여, 생성된 평가 결과 HTML 리포트를 함께 제출하세요.
이는 자기 결과물을 **증거 기반으로 스스로 검증**하는 Full Cycle 역량의 일부입니다.

- Agent가 생성한 HTML 리포트(`..._에이전트(X팀).html`)를 그대로 포함합니다.
- 리포트에서 드러난 누락·미충족 항목(미구현 요구사항, 실패 테스트, 시연 미입증 등)은 제출 전에 보완하세요.
- 보완이 어려운 항목은 결과보고서에 사유를 남기는 것을 권장합니다.

> 자체 평가 리포트는 "제출 시점 기준"으로 생성된 최신본을 넣으세요. 최종 제출물과 내용이 어긋나지 않도록 합니다.
