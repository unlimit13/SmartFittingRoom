**프로젝트명**: LG Styler Smart Fitting Room — On-Device AI 기반 실시간 패션 추천 시스템

## 요구사항 목록

> - 요구사항은 **기능 요구사항(`FR-01`…)** 과 **비기능 요구사항(`NFR-01`…)** 으로 구분한다.
> - **기능 요구사항**은 사용자가 관찰 가능한 동작이므로 시연 영상 타임스탬프를 표기한다.
>   내부 파이프라인 단계처럼 영상으로 시연이 불가한 항목은 `N/A (사유)`로 표기한다.
> - **비기능 요구사항**은 성능·제약 등 품질 속성으로, 시연 타임스탬프 대신 측정·코드 검증으로 확인한다.
> - 시연 타임스탬프는 시연 영상 촬영 후 확정하며, 현재는 예정 시간으로 표기한다.

## 요구사항 상세 설명

### FR-01: 웹캠 라이브 피드 표시

웹 브라우저에서 RPi5의 카메라 영상을 실시간으로 확인할 수 있어야 한다.
Flask의 MJPEG 스트리밍 방식(`/detection_feed`)을 사용하며, 네트워크 내 어느 기기 브라우저에서도 접속 가능하다.
이 스트림에는 포즈 스켈레톤·의류 영역 박스·손 커서 등의 오버레이가 함께 그려진다 (FR-02·FR-03 참조).

### FR-02: 의류 영역 자동 감지 및 시각화

`detector.py`의 MediaPipe Pose(`static_image_mode=True`, `model_complexity=0`)가 카메라 프레임에서
33개 랜드마크를 검출한다. visibility ≥ 0.5인 랜드마크들의 좌표 범위로 사람 바운딩 박스를 산출하고,
신뢰도(confidence)는 필수 관절(코·양 어깨·양 엉덩이, 인덱스 0/11/12/23/24) visibility의 평균으로 계산한다.
산출된 바운딩 박스를 수직 비율로 분할하여 의류 크롭을 생성한다.
- 상의(tops): y_top ~ y_top + 45%
- 하의(bottoms): y_top + 40% ~ y_top + 80%
- 신발(shoes): y_top + 75% ~ y_bottom

MediaPipe Pose는 프레임당 단일 인물을 추적한다 (필수 관절이 모두 보이지 않으면 미검출 처리).
라이브 피드(`detection_feed`)에는 스켈레톤·의류 영역 박스 오버레이가 그려지며, `/overlay_toggle` POST로
전역 표시 상태(`_show_overlay`)를 on/off 할 수 있다. 감지 결과(사람 바운딩 박스 + 신뢰도 +
상의/하의/신발 영역)는 `/recommend` 추천 시점에 그려진 주석 이미지(`annotated_b64`)로 응답에 포함되어 UI에 표시된다.

### FR-03: 손 인식 커서 입력

MediaPipe Pose(`pose.py`)가 카메라 프레임에서 왼손(랜드마크 19/17/21/15 우선순위)·오른손
(20/18/22/16 우선순위)의 좌표를 추출해 프레임 크기 기준 `[0,1]`로 정규화하고, `/pose_poll` (GET)이
`lw`/`rw` 필드로 반환한다. 프론트엔드는 이를 화면 좌표로 매핑해 좌/우 두 개의 커서(`cursor-l`/`cursor-r`)로
렌더한다.

커서가 활성 버튼(`추천 받기`, `◀`/`▶`, `입어보기` 등) 위에 `DWELL_MS(1초)` 동안 머무르면 dwell 진행바가
차오르고, 100% 도달 시 해당 버튼의 액션이 트리거된다(마우스 클릭과 동일한 dispatch). 손 커서는 이 시스템의
공용 입력 수단으로, 추천 발동(FR-08)·세트 전환(FR-11)·가상 피팅 실행(FR-10)에 모두 사용된다.

> 참고: `pose.py`에는 필수 관절이 화면 중앙 존에 일정 시간 유지되면 발동하는 존/홀드 트리거 로직
> (`in_zone`·`hold_pct`·`triggered`·`reset()`)이 남아 있으나, 현재 프론트엔드는 이를 자동 추천 트리거로
> 사용하지 않는다. `in_zone`은 "전신이 인식되었으니 손으로 버튼을 눌러보세요" 안내 힌트로만 쓰인다.

### FR-04: CLIP 이미지 임베딩 추출

감지된 의류 크롭 이미지를 224×224로 리사이즈하고 CLIP 전처리(정규화)를 적용한 뒤,
CLIP ViT-B/32 (ONNX) 모델로 추론한다.
출력 시퀀스의 CLS 토큰(768-dim)에 `visual_projection.npy` (512×768) 행렬의 전치를 곱해
512차원 벡터로 변환하고, L2 정규화하여 최종 임베딩을 생성한다.
이 벡터가 FAISS 유사도 검색의 쿼리로 사용된다.

### FR-05: FAISS 유사도 검색

사전 빌드된 무신사 의류 118장 FAISS 인덱스에서 CLIP 이미지 임베딩 기반 유사 후보를 검색한다.
`anchor_category` 카테고리로 필터링하여 Top-20을 반환한다 (over-fetch k×3 후 카테고리 필터).
검색 인덱스는 시스템 시작 시 메모리에 로드되며 런타임 중 재빌드하지 않는다.

### FR-06: 한국어 텍스트 임베딩 지원

`text_encoder.py`에서 `jhgan/ko-sroberta-multitask` (ONNX) 모델로 한국어 입력을 인코딩한다.
- 토크나이저(max_length=128) → ONNX 추론 → Mean Pooling → L2 정규화 768-dim 벡터
- `recommender.py`의 `recommend_outfit()`에서 `text_query`가 있을 경우 `TextEncoder.encode()`를 호출하여 `text_vec`를 생성한다.
- `text_vec`는 `Reranker.rerank()`에 전달되어 최종 점수 산출에 반영된다: `α×clip_sim + β×text_sim + γ×color_compat`
  - 텍스트 제공 시 가중치: α=0.4, β=0.4, γ=0.2
  - 텍스트 없을 시 가중치: α=0.6, β=0.0, γ=0.4

### FR-07: 색상 팔레트 추출

OpenCV K-means 알고리즘을 사용해 `anchor_category`(상의 또는 하의)의 크롭 영역에서 지배 색상 3가지를 추출한다.
추출된 색상은 추천 응답의 `palette` 필드로 반환되어 UI에 시각적으로 표시된다.
추출된 팔레트는 색상 호환성 점수(`reranker.py`의 `_color_compat()`, HSV 기반)로 변환되어
최종 추천 순위의 γ(색상 호환성) 가중치에 실제로 반영된다 (FR-08 참조).

### FR-08: 손 커서 추천 트리거 + 혼합 점수 기반 코디 세트 최대 3개 생성

사용자가 손 커서로 `추천 받기` 버튼을 dwell 트리거(FR-03)하면 `/recommend`가 호출된다.
`anchor_category`(상의 또는 하의) 크롭의 CLIP 이미지 임베딩 벡터로 앵커 카테고리를 검색하고,
각 앵커 상품을 중심으로 스냅(큐레이션 코디) 매칭 또는 유사도 폴백으로 나머지 슬롯을 채워
**최대 3개(`NUM_CANDIDATES=3`)의 코디 세트**를 반환한다.

추천 알고리즘 (`recommender.py: recommend_outfit()`):
1. `anchor_category` 크롭 → CLIP 임베딩(512-dim) → 앵커 필드(상의기준=`tops`, 하의기준=`bottoms`) FAISS 검색(카테고리 필터, Top-20 후보)
2. Reranker 혼합 점수 `α×clip_sim + β×text_sim + γ×color_compat`로 앵커 상품 **Top-3** 선정 (`anchor_items`)
3. 각 앵커 상품마다 코디 세트를 구성:
   - 해당 상품의 `snap_id`에 큐레이션 스냅 코디가 있으면 나머지 슬롯(반대 상·하의, 신발)을 스냅에서 가져옴
   - 스냅이 없으면 유사도 검색 폴백: 반대 필드·신발을 각각 Top-3 검색해 i번째 후보를 사용
4. 하의기준(`anchor_category="bottoms"`)일 때 `outfit["bottoms"]`는 빈 리스트(착용 하의는 사용자 캡처라 DB 상품으로 표시하지 않음)이며, `result["bottoms_crop"]`에 캡처 크롭을 담아 반환
5. 검색 후보가 3개 미만이면 그 수만큼만 세트를 반환 (인덱스 에러 없음)

앵커 상품이 혼합 점수 내림차순으로 선정되므로, 반환되는 `outfits`도 추천도가 높은 순서로 정렬된다
(세트 1 = 최고 추천). 이 순서가 FR-11의 세트 번호(1-2-3)로 이어진다.

API 응답 구조 (`outfits`는 최대 3개):
```json
{
  "detected": true,
  "palette": ["#RRGGBB", ...],
  "outfits": [
    {
      "tops":    [{"product_id": "...", "name": "...", "url": "...", "image_path": "...", "qr_b64": "..."}],
      "bottoms": [...],
      "shoes":   [...]
    }
    // ... 최대 3개
  ],
  "annotated_b64": "...",
  "captured_top_b64": "...",
  "captured_bottom_b64": "...",
  "elapsed_ms": 850
}
```

결과는 UI 추천 패널에 코디 카드로 표시하며, 각 상품에 QR코드가 포함된다.

### FR-09: 무신사 상품 QR코드 생성

추천 코디 세트를 구성하는 각 상품(상의·하의·신발)에 대해 무신사 상품 URL이 인코딩된 QR코드를
`qrcode` 라이브러리로 생성하여 base64 PNG(`qr_b64`)로 응답에 포함하고 UI에 표시한다.
스마트폰으로 QR코드를 스캔하면 해당 무신사 상품 페이지로 바로 이동할 수 있다.

### FR-10: 가상 피팅(Virtual Try-On)

추천된 상의/하의를 실제 착용자에게 합성한다. 백엔드는 환경 변수 `VTON_BACKEND`로 선택한다.

- **기본 (`api`)**: `fal-ai/fashn/tryon v1.6` 외부 클라우드 API 사용 (`tryon.py`, `FAL_KEY` 필요, `fal_client==1.0.0`).
- **온디바이스 (`ondevice`)**: 벤더링된 Mobile-VTON 파이프라인을 클라우드 호출 없이 RPi5에서 실행 (`tryon_ondevice.py`). 인물 이미지가 로컬 파일 경로로 처리된다. 디노이저를 4대 RPi5에 공간 분산하는 부분은 **NFR-03**이 담당하며, 클러스터 구성·환경 변수는 RUN.md §9 참조.

두 백엔드는 동일한 인터페이스(`upload_frame` / `run_tryon_stream` / `fetch_b64`)를 노출하여 `app.py`가 동일하게 호출한다.

순차 합성 전략:
1. 추천 발동 시 현재 카메라 프레임을 스토리지에 비동기 업로드 (`/recommend` 호출과 병렬)
2. 상의 피팅 요청 → 합성 결과 이미지 반환 → 하의 피팅 시 상의 결과를 인물 이미지로 재사용
3. `/tryon` (POST, SSE): `top_image_path` 또는 `bottom_image_path`(또는 둘 다)를 전달하면
   `{"step": "tops"|"bottoms", "result_b64": "...", "mime": "image/jpeg"}` 이벤트를 스트리밍으로 반환

의존성:
- 기본 백엔드는 `FAL_KEY` 환경 변수 필요 (`fal_client==1.0.0`)
- `top_image_path` / `bottom_image_path`는 `data/musinsa_db/` 하위 상대 경로
- 인물 이미지는 `/recommend` 호출 시 자동 업로드되므로 반드시 추천 후 `/tryon`을 호출해야 한다.

사용자는 손 커서로 `입어보기` 버튼을 dwell 트리거해 미리 받아둔(또는 로딩 중인) 가상 피팅 결과를 띄운다.

### FR-11: 손 커서 세트 네비게이션

FR-08이 생성한 최대 3개 코디 세트를 손 인식 커서로 탐색한다.
세트는 추천도가 높은 순서대로 1-2-3 번호를 가지며(FR-08의 정렬 순서), 프론트엔드 상태 `S.outfits[S.idx]`로
현재 세트를 렌더한다.

- 손 커서로 `◀`(이전)·`▶`(다음) 버튼을 dwell 트리거하면 `navigate(±1)`이 호출되어
  `S.idx = (S.idx + dir + N) % N`로 세트 인덱스를 순환(wrap-around) 전환한다.
- 전환 시 현재 세트의 상의/하의/신발 카드와 QR이 다시 렌더되고,
  하단 배지(`results-badge`)에 현재 세트 번호가 `현재/전체`(예: `2 / 3`) 형태로 표시된다.

세트 전환·번호 표시 로직은 프론트엔드(`index.html`)에 구현되어 시연으로 확인한다.
그 기반이 되는 "최대 3세트가 추천도 내림차순으로 준비된다"는 백엔드 계약은
`tests/test_recommender_outfit.py`(FR-08)로 검증한다.

### NFR-01: 전체 응답 시간 2초 이내

카메라 입력과 텍스트 입력이 모두 제공된 조건에서 `/recommend` API의 전체 응답 시간이 2,000ms 이하여야 한다.
주요 파이프라인 예상 시간: MediaPipe Pose 50ms + CLIP 이미지 300ms + FAISS 5ms + ko-sroberta 200ms + 리랭킹 5ms ≈ 560ms.

### NFR-02: AI 추론 On-Device 동작 (가상 피팅 제외)

의류 감지(MediaPipe Pose), 이미지 임베딩(CLIP), 텍스트 임베딩(ko-sroberta), 유사도 검색(FAISS) 모두
RPi5 로컬에서 실행된다(CLIP·ko-sroberta·FAISS는 ONNX Runtime, 의류 감지는 MediaPipe Pose).
AI 추론 파이프라인에서 외부 API 또는 클라우드 서비스를 호출하지 않는다.
(YOLOv8n은 런타임 의류 감지에 쓰이지 않으며, 로컬 데이터 준비 단계의 크롤 이미지 사람 필터(`crawl_musinsa.py`)에서만 사용된다.)

단, 가상 피팅 기능(FR-10)은 기본 백엔드에서 `fal-ai/fashn/tryon v1.6` 외부 API를 사용하며 `FAL_KEY` 환경 변수가 필요하다.
가상 피팅은 선택적 기능으로, `FAL_KEY` 미설정 시 `/tryon` 엔드포인트를 제외한 나머지 시스템은 정상 동작한다.

### NFR-03: 분산 가상 피팅 추론 (Mobile-VTON spatial-parallel 클러스터)

가상 피팅 디노이저(1024×768)는 단일 RPi5에서 OOM이라, 활성화 시(`VTON_BACKEND=ondevice`)
여러 RPi5에 **H축(row) 방향으로 활성화 텐서를 분산**하여 추론한다. 가중치는 모든 rank에
복제되고, 각 rank는 모든 `[B,C,H,W]` 텐서의 연속 row-band `[h0:h1)`만 소유한다.
코드는 `src/ondevice_vton/parallel/`에 있다. 이 분산 추론이 온디바이스 가상 피팅(FR-10)을 뒷받침한다.

역할 분담(rank = 위→아래 순서, rank0 = 앱 구동 보드):

| rank | row-band | 담당 |
|---|---|---|
| rank0 | 최상단 행 | 앱 서빙 + 디노이저 상단 band + 분산 런처(rsync/실행) |
| rank1..N-1 | 이하 연속 band | 디노이저 해당 band 추론 (peer) |

- **분할 규칙**(`sp_common.band_bounds`): H를 가능한 한 균등 분할, 나누어떨어지지 않으면
  앞 rank가 +1을 가져간다. 모든 rank가 서로의 band를 동일하게 계산하도록 **결정적**이다.
  2-Pi에서는 `SP_BAND_FRAC0`로 rank0 비율을 조정해 비균등 분할도 가능하다(RAM 편차 대응).
- **통신이 필요한 연산만 동기화**: kernel>1 conv는 경계 `(k-1)//2`행 halo 교환(`sp_conv`),
  GroupNorm은 그룹별 (합·제곱합·개수) all-reduce(`sp_groupnorm`), self-attention은 K/V
  all-gather(`sp_attention`). 그 외(1×1 conv·FFN·SiLU·cross-attention 등)는 band-독립.
- **동치성**: 입구에서 한 번 scatter, 출구에서 한 번 gather하므로 결과는 단일 Pi와 동치다.

검증: row-band 분할(`band_bounds`)의 결정성·연속성·무중첩 전체 커버리지와 비균등 분할
(`SP_BAND_FRAC0`)은 `tests/test_sp_common.py`(NFR-03)로 단위 검증한다. 분할 로직은 torch
비의존 모듈(`src/ondevice_vton/parallel/sp_bands.py`, `sp_common`이 그대로 re-export)로
분리되어 있어, torch가 없는 메인 스위트(`requirements.txt`)에서도 실제로 실행·통과한다.
다중 Pi 분산 실행 자체는 클러스터 시연으로 확인한다(RUN.md §9·§9-5,
`src/ondevice_vton/parallel/PROGRESS.md`·`OPTIMIZATION_HISTORY.md`).

## 시연 시나리오

```
[시연 시작]
1. 웹 브라우저에서 http://<RPi1_IP>:5000 접속
2. 카메라 라이브 피드 확인 (흰 티셔츠 + 청바지 착용 상태)
3. MediaPipe Pose 기반 사람/의류 영역 바운딩 박스 + 스켈레톤 오버레이 자동 표시 확인 (FR-02)
4. 색상 팔레트 추출 결과 확인 (FR-07)
5. 손 커서를 [추천 받기] 버튼 위에 dwell(1초) → 추천 발동 → 코디 세트 최대 3개 생성 (FR-03·FR-08)
6. 텍스트 입력: "맑은 날 여자친구와 데이트하려고 해. 추천해줘."
   → 추천 결과 순위 변화 확인 (FR-06)
7. 추천 상품 QR코드 스캔 → 무신사 상품 페이지 이동 확인 (FR-09)
8. 손 커서로 ◀/▶ 버튼 dwell → 다음/이전 코디 세트 전환, 하단 세트 번호 확인 (FR-11)
9. 손 커서로 [입어보기] 버튼 dwell → 상의/하의 가상 피팅 합성 결과 표시 확인 (FR-10)
10. (NFR-03) `VTON_BACKEND=ondevice` — 4대 RPi 공간 분산 가상 피팅 추론 시연
[시연 종료]
```
