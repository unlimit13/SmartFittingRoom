## 요구사항 상세 설명

### R-01: 웹캠 라이브 피드 표시

웹 브라우저에서 RPi5의 카메라 영상을 실시간으로 확인할 수 있어야 한다.
Flask의 MJPEG 스트리밍 방식(`/video_feed`)을 사용하며, 네트워크 내 어느 기기 브라우저에서도 접속 가능하다.

### R-02: 의류 영역 자동 감지

YOLOv8n (ONNX) 모델이 카메라 영상에서 사람(COCO class 0, confidence ≥ 0.5)을 감지한다.
감지된 사람 바운딩 박스를 수직 비율로 분할하여 의류 크롭을 생성한다.
- 상의(tops): y_top ~ y_top + 45%
- 하의(bottoms): y_top + 40% ~ y_top + 80%
- 신발(shoes): y_top + 75% ~ y_bottom

사람이 여러 명 감지될 경우 가장 큰 바운딩 박스(면적 기준)를 사용한다.
감지 결과(사람 바운딩 박스 + 신뢰도)는 라이브 피드에 오버레이되어 표시된다.

### R-03: CLIP 이미지 임베딩 추출

감지된 의류 크롭 이미지를 224×224로 리사이즈하고 CLIP 전처리(정규화)를 적용한 뒤,
CLIP ViT-B/32 (ONNX) 모델로 추론한다.
출력 시퀀스의 CLS 토큰(768-dim)에 `visual_projection.npy` (768×512) 행렬을 곱해
512차원 벡터로 변환하고, L2 정규화하여 최종 임베딩을 생성한다.
이 벡터가 FAISS 유사도 검색의 쿼리로 사용된다.

### R-04: 색상 팔레트 추출

OpenCV K-means 알고리즘을 사용해 `anchor_category`(상의 또는 하의)의 크롭 영역에서 지배 색상 3가지를 추출한다.
추출된 색상은 UI에 시각적으로 표시된다.
색상 호환성 로직(`reranker.py`의 `_color_compat()`)은 구현되어 있으나 현재 추천 파이프라인에서 미사용.

### R-05: FAISS 유사도 검색

사전 빌드된 무신사 의류 118장 FAISS 인덱스에서 CLIP 이미지 임베딩 기반 유사 후보를 검색한다.
`anchor_category` 카테고리로 필터링하여 Top-20을 반환한다 (over-fetch k×3 후 카테고리 필터).
검색 인덱스는 시스템 시작 시 메모리에 로드되며 런타임 중 재빌드하지 않는다.

### R-06: 한국어 텍스트 임베딩 지원

`text_encoder.py`에서 `jhgan/ko-sroberta-multitask` (ONNX) 모델로 한국어 입력을 인코딩한다.
- 토크나이저(max_length=128) → ONNX 추론 → Mean Pooling → L2 정규화 768-dim 벡터
- `recommender.py`의 `recommend_outfit()`에서 `text_query`가 있을 경우 `TextEncoder.encode()`를 호출하여 `text_vec`를 생성한다.
- `text_vec`는 `Reranker.rerank()`에 전달되어 최종 점수 산출에 반영된다: `α×clip_sim + β×text_sim + γ×color_compat`
  - 텍스트 제공 시 가중치: α=0.4, β=0.4, γ=0.2
  - 텍스트 없을 시 가중치: α=0.6, β=0.0, γ=0.4

### R-07: 혼합 점수 기반 코디 세트 1개 표시

`anchor_category` 크롭의 CLIP 이미지 임베딩 벡터로 tops/bottoms/shoes 카테고리를 **독립적으로** 검색하여 1개의 코디 세트를 반환한다.

추천 알고리즘:
1. `anchor_category` 크롭 → CLIP 임베딩(512-dim) → 동일 벡터로 tops/bottoms/shoes 각각 FAISS 검색(카테고리 필터, Top-20 후보)
2. 각 카테고리 후보에 대해 Reranker가 혼합 점수 산출: `α×clip_sim + β×text_sim + γ×color_compat`
3. 카테고리별 혼합 점수 1위 상품을 선정 → 코디 세트 구성

API 응답 구조:
```json
{
  "detected": true,
  "palette": ["#RRGGBB", ...],
  "outfits": [
    {
      "tops":    [{"product_id": "...", "name": "...", "url": "...", "image_path": "...", "qr_b64": "..."}],
      "bottoms": [...],
      "shoes":   [...]
    }
  ],
  "elapsed_ms": 850
}
```

결과는 UI 추천 패널에 1개 코디 카드로 표시하며, 각 상품에 QR코드가 포함된다.

### R-08: 무신사 상품 QR코드 생성

추천된 Top-3 각 상품에 대해 무신사 상품 URL이 인코딩된 QR코드를 `qrcode` 라이브러리로 생성하여 UI에 표시한다.
스마트폰으로 QR코드를 스캔하면 해당 무신사 상품 페이지로 바로 이동할 수 있다.

### R-09: 전체 응답 시간 2초 이내

카메라 입력과 텍스트 입력이 모두 제공된 조건에서 `/recommend` API의 전체 응답 시간이 2,000ms 이하여야 한다.
주요 파이프라인 예상 시간: YOLO 50ms + CLIP 이미지 300ms + FAISS 5ms + ko-sroberta 200ms + 리랭킹 5ms ≈ 560ms.

### R-10: AI 추론 On-Device 동작 (가상 피팅 제외)

의류 감지(YOLOv8n), 이미지 임베딩(CLIP), 텍스트 임베딩(ko-sroberta), 유사도 검색(FAISS) 모두
RPi5 로컬에서 ONNX Runtime으로 실행된다. AI 추론 파이프라인에서 외부 API 또는 클라우드 서비스를 호출하지 않는다.

단, 가상 피팅 기능(R-13)은 `fal-ai/fashn/tryon v1.6` 외부 API를 사용하며 `FAL_KEY` 환경 변수가 필요하다.
가상 피팅은 선택적 기능으로, `FAL_KEY` 미설정 시 `/tryon` 엔드포인트를 제외한 나머지 시스템은 정상 동작한다.

### R-12: 포즈 기반 자동 추천 트리거

MediaPipe Pose(model_complexity=0)가 카메라 프레임에서 관절을 추적한다.
화면 중앙 존(ZONE = 좌 22%~우 78%, 상 3%~하 97%)에 필수 관절(코·양 어깨·양 엉덩이) 5개가
모두 들어온 상태로 3초(`HOLD_SEC = 3.0`) 유지되면 추천이 자동 발동된다.

- `/pose_poll` (GET): 현재 존 진입 여부(`in_zone`), 진행률(`hold_pct`), 트리거 여부(`triggered`), 비활성 여부(`disabled`), 손목 좌표(`rw`) 반환
- `/pose_reset` (POST): 트리거 후 비활성화된 존을 재활성화
- 트리거 발동 후 존은 자동으로 비활성(`disabled: true`) 상태가 되며, 사용자가 취소하거나 재시작할 때까지 재트리거되지 않는다.
- `detection_feed` 스트림에 존 사각형·홀드 진행바·잔여 시간이 오버레이로 표시된다.

### R-13: 가상 피팅(Virtual Try-On)

`fal-ai/fashn/tryon v1.6` 외부 API를 사용하여 추천된 상의/하의를 실제 착용자에게 합성한다.

순차 합성 전략:
1. 추천 발동 시 현재 카메라 프레임을 fal 스토리지에 비동기 업로드 (`/recommend` 호출과 병렬)
2. 상의 피팅 요청 → 합성 결과 이미지 반환 → 하의 피팅 시 상의 결과를 인물 이미지로 재사용
3. `/tryon` (POST, SSE): `top_image_path` 또는 `bottom_image_path`(또는 둘 다)를 전달하면
   `{"step": "tops"|"bottoms", "result_b64": "...", "mime": "image/jpeg"}` 이벤트를 스트리밍으로 반환

의존성:
- `FAL_KEY` 환경 변수 필요 (`fal_client==1.0.0`)
- `top_image_path` / `bottom_image_path`는 `data/musinsa_db/` 하위 상대 경로
- 인물 이미지는 `/recommend` 호출 시 자동 업로드되므로 반드시 추천 후 `/tryon`을 호출해야 한다.

### R-11: 분산 Edge AI 파이프라인 (Phase 2 — 미구현)

> **현재 상태**: Phase 1 단일 RPi 파이프라인 완성. Phase 2 워커 서버(`worker_embed.py`, `worker_search.py`, `worker_aggregate.py`)는 미구현.

계획된 역할 분담:

| 보드 | IP | 담당 역할 |
|---|---|---|
| RPi1 | 10.56.130.185 | 카메라 캡처 + YOLOv8n 감지 + 웹 UI 서빙 |
| RPi2 | 10.56.130.178 | CLIP 이미지 임베딩 + ko-sroberta 텍스트 임베딩 워커 |
| RPi3 | 10.56.130.182 | FAISS 검색 + 텍스트 리랭킹 워커 |
| RPi4 | 미확인 | 결과 집계 + 색상 호환성 + 최종 응답 |

보드 간 통신은 HTTP REST. Phase 1과 동일한 추천 결과가 분산 환경에서도 재현되어야 한다.
